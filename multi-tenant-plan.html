Multi-tenant Plan — Platform Admin vs. Per-School Admin

Context

The platform runs single-tenant today on huaxia.mschool.com.tw. The "admin panel" does two jobs at once: it manages platform-level data (schools, global story library, audit log, R2 browser, system settings) and school-level data (users, classes, recordings for the one school). If a second school is added — e.g. fongshan.mschool.com.tw — that conflation forces either a duplicated panel or a confused permission model.

Goal: design the cleanest path to a clean split, without disrupting Huaxia.

  • mschool.com.tw/admin = platform admin (schools, billing, global library, system).
  • <school>.mschool.com.tw/admin = school admin (their users, classes, audit log, recordings).

The data model is already 90% set up for this. classes.schoolId is a foreign key (src/lib/db/schema.ts:185), assignments → classes, recordings → assignments. School-scoping is implicit in joins today; formalising it is mostly adding tenant context to the request and narrowing one role check.

This document is a think-piece. No code changes are committed by reading it; it's here so the architecture decision is on paper before we choose to act.

Recommended Architecture arch

Single Next.js deployment, single Postgres, wildcard cert *.mschool.com.tw. src/middleware.ts parses Host, looks up the subdomain against a new schools.slug column, and injects an x-tenant-school-id header on the forwarded request. Every server component / API route reads the header via a getTenantSchool() helper.

Per-school admin authority lives on school_memberships.role, not on users.role. Rationale: a user is a teacher in one school and an admin in another; expressing that on the membership row keeps things flat.

Critical companion move: rename the admin value in the users.role enum to platform_admin in the same migration. The ~24 existing role === 'admin' checks in API routes today silently grant cross-school access; if we leave the word "admin" ambiguous, every refactor risks a tenant leak. Force the rename so the word unambiguously means "global root."

Role + Permission Model

Two-axis model:

  • users.role: student | teacher | platform_admin
  • school_memberships.role: member | admin (new enum, default member)

Translation of today's admin checks:

TodayAfter
role === 'admin' for platform-only resources (schools, global stories, settings, R2) role === 'platform_admin' only
role === 'admin' for a school-scoped resource (users, classes, audit log for one school) platform_admin or membership.role = 'admin' on the tenant school
role === 'admin' short-circuit in userCanManageClass() (src/lib/auth/class-access.ts:19) platform_admin → all classes;
school-admin → classes whose schoolId matches one of their admin memberships;
teacher branch unchanged.

/api/admin/* becomes the platform admin surface. Per-school admin can live under the same /api/admin/* tree, but every handler asserts getTenantSchool() and filters by it; routes that have no tenant (e.g. "create school") require platform_admin.

Tenant Context Propagation

Middleware-injected header. Not session, not async context.

  1. src/middleware.ts reads Host, strips .mschool.com.tw, looks up schools.slug = subdomain (in-memory TTL cache, ~60s), sets x-tenant-school-id on the forwarded request via NextResponse.next({ request: { headers } }).
  2. Server side: getTenantSchool() reads the header via headers() from next/headers (server components) or request.headers.get(...) (route handlers).

Failure modes

HostBehavior
localhost / localhost:3000Null tenant; routes that require one return 404. Dev override via X-Dev-School header (gated by NODE_ENV !== 'production').
Apex mschool.com.twNull tenant; only platform-admin and marketing render.
Unknown subdomainRender a clean "school not found" page. Never fall through to a random tenant. Same page whether the slug is unknown or the school is inactive (don't leak enumeration).
DB lookup failsFail closed (403). Do not silently treat as platform.

Schema Deltas (minimum set) schema

  1. schools.slug varchar(63) unique not null — RFC-1035 label rules (lowercase, digits, hyphens). Backfill huaxia for the existing row, then enforce NOT NULL.
  2. New enum school_membership_role = ['member', 'admin']. Add school_memberships.role (default member).
  3. Rename enum value users.role.admin → platform_admin (ALTER TYPE ... RENAME VALUE). Update ~24 code call sites in the same PR.
  4. audit_logs.school_id uuid null references schools(id), indexed. Nullable because platform-level events ("school created") have no school.
  5. Stories / reading passages: add nullable school_id but don't start scoping queries on it. Zero-cost optionality; the column is there waiting if we later decide to scope.

Cookie Strategy

Host-only cookies (status quo) — do NOT set Domain=.mschool.com.tw.

COOKIE_OPTIONS in src/lib/auth.ts:171 already omits domain, so the session cookie is host-bound. Keep it that way:

  • Smaller blast radius if a session is stolen.
  • Student tokens (/s/[token]) are issued to a user globally; allowing them to authenticate on a subdomain the student doesn't belong to would be a leak.
  • Tenancy becomes a hard wall, not a convention.

Platform-admin DX (jumping between schools without re-logging in) gets a one-shot signed redirect token: a "switch school" button on mschool.com.tw/admin mints a 5-second single-use token, redirects to <school>.mschool.com.tw/admin/_resume?token=..., which sets a fresh session cookie scoped to that subdomain. Same pattern as Google's account chooser.

Code-comment the cookie config loudly. "Convenience" requests to add domain=.mschool.com.tw later would collapse the tenant wall instantly.

DNS + Cert + Deploy

  • Wildcard DNS A record *.mschool.com.tw → VPS IP.
  • Wildcard ACME cert via DNS-01 challenge (HTTP-01 cannot issue wildcards). Caddy + Cloudflare DNS module or Coolify's wildcard support. Confirm the registrar exposes an API token for DNS-01 before committing to the timeline.
  • No changes needed in next.config.mjs.
  • Single Next.js process serves every subdomain; no per-tenant deploy.

Migration in Phases phased

Each phase ships independently. Phase 1 is a no-op for end users.

PhaseWhat shipsEnd-user impact
1 Add schools.slug, backfill huaxia. Middleware resolves Host → schoolId and sets the header, but no route reads it yet. None. Staging subdomains used to verify the header lands.
2 Membership role column + audit_logs.school_id. Migrations only; defaults populate everyone as member. None.
3 Rename admin → platform_admin. Single PR updates all ~24 call sites; same deploy runs ALTER TYPE. The existing one admin user becomes platform_admin. None (behavior identical, name is now honest). Fragile — green-deploy the code before the migration runs.
4 userCanManageClass, accessibleClassIds, and the user/class/audit-log admin APIs grow the membership-role branch. Dead code until phase 5. None.
5 Wildcard DNS + cert active. Create fongshan school + slug. Promote a teacher to school_memberships.role = 'admin' for Fongshan. Both panels go live. Fongshan onboards. Huaxia URL unchanged.
6 Audit every API route for "does this respect the tenant header?" Add CI lint that fails if a school-scoped route doesn't call getTenantSchool(). Ship the cross-school redirect-token flow. None directly; safety net + platform-admin DX.

Decision Matrix

DecisionRecommendationWhy
Database topology Single DB with school_id columns Schema is FK-clean already. Per-tenant DBs would force migrations × N forever and complicate platform-level reporting.
URL shape Subdomain (huaxia.mschool.com.tw) Cleaner cookie/CSP boundary, no rewriting of student QR codes or email links, visually distinct in the URL bar.
Session cookie scope Host-only Smaller blast radius on session theft; student tokens stay tenant-bound; platform-admin DX solved with the one-shot redirect token.
Admin role location Membership-level (school_memberships.role) A user can be a teacher in one school and admin in another. Expresses cleanly without a parallel school_admins table.

What We Punt deferred

  • Custom per-school domains (e.g. fongshan.edu.tw instead of subdomain). Needs per-tenant cert provisioning + Host allowlist. Add when a school asks.
  • Per-tenant databases. Operational cost is huge. Only revisit for data-residency requirements.
  • Story / passage library scoping. Today every school sees every story. Ship the nullable school_id column for future-us but don't scope queries.
  • Per-school theming / branding (logo, colors). Trivial to add later — schools.logoUrl, schools.primaryColor.
  • Cross-school class sharing. If two schools want to share a teacher, that teacher gets two memberships and switches subdomains. No native sharing.
  • Audit-log unification. Platform admin will want a global audit view that ignores tenant header. Fine, but document it so it isn't an accidental leak.

Risks Worth Naming risk

  • Silent tenant leak. If a route forgets getTenantSchool() after phase 5, it returns cross-school data without complaint. Hence the phase-6 lint.
  • Cookie scope mistake. Adding domain=.mschool.com.tw "for convenience" later would collapse the tenant wall instantly. Comment loudly in auth.ts.
  • Migration ordering. Phase 3 (enum rename) is fragile; running ALTER TYPE before the code is deployed will throw 500s on every role === 'admin' check. Use a green-deploy where the new code rolls out before the migration runs.
  • Subdomain enumeration. unknown.mschool.com.tw should not leak whether a school exists. Same "school not found" page whether the slug is unknown or the school is inactive.
  • Platform-admin DX entropy. Constant re-login is annoying; without the redirect-token flow, the demand for wildcard cookies will eventually arrive. Build the redirect flow as part of phase 6, not "later".

TL;DR

  1. One Next.js app + one Postgres + wildcard cert. Subdomain → schoolId via middleware-injected header.
  2. Rename admin → platform_admin. Add school_memberships.role. School admins live on the membership, not on the user.
  3. Cookies stay host-only. Platform admin gets a one-shot redirect token to hop schools.
  4. Six small phases; phase 1 is invisible to users; phase 5 is when Fongshan goes live.
  5. Punt custom domains, per-tenant DBs, library scoping, and theming until someone asks.
