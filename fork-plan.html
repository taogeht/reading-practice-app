Fork Plan — Spawn a New School Deployment

Context

The current recording-homework-app is functionally generic (auth, recording loop, spelling, attendance, syllabus tracking, AI-generated reading passages), but its content layer is hardwired to Macmillan Language School in Kaohsiung — specifically to the American Family & Friends (AF&F) curriculum. The goal is to stand up a second deployment for a different school using everything we've built so far, then let the two codebases evolve independently. The new school targets the same demographic (Mandarin-speaking ESL kids in Taiwan, ages 6–10) but with a different curriculum.

Decision summary (already made):

  • Hard fork — new git repo, no upstream merges, fully independent.
  • Fully separate infrastructure — new Postgres, new R2 bucket, new GCP TTS project, new domain, new Coolify app, new API keys.
  • Only the curriculum changes. Cultural prompt cues (night market, scooter, temples) and the mandarinTranslation column stay.
  • Credential rotation for the leaked keys in .env.production.backup handled separately (out of scope here).

Strategy at a Glance

Four work streams, roughly sequential, with one parallelizable subtree:

  1. Provision — stand up the new infrastructure and verify each service in isolation before touching code.
  2. Clone & rebrand — fork the repo, set new identity, scrub Macmillan references from prompts and metadata.
  3. Replace curriculum — swap the AF&F-bound data (books, levels, vocabulary, units, workbooks). This is the bulk of the work.
  4. Deploy & verify — push to Coolify, run a full smoke test of the reading loop end-to-end with the new curriculum.

Phase 4 — Schema & Seed Cleanup data

4.1 Decide the fate of AF&F-named columns

The schema has AF&F-coded enums/columns in src/lib/db/schema.ts (afFLevel, afFUnit, comments referencing AF&F). For a clean fork:

  1. Create a migration that renames af_f_level → curriculum_level and af_f_unit → curriculum_unit (or whatever naming fits the new curriculum).
  2. Update the corresponding enum values to match the new curriculum's levels.
  3. Update all TypeScript references via project-wide rename.

Alternative for speed: keep the names, just feed the new curriculum's labels into them. Uglier but reduces churn. Recommendation: rename, since the fork is meant to develop independently and the cosmetic debt will compound otherwise.

4.2 Fresh seed data

Edit src/lib/db/seed.ts:

  • Demo school name "Riverside Elementary School" → new school
  • Remove the three sample stories with stub URLs (already cleaned out of the current DB; re-clean for the new one)
  • Admin email/password — set to a known dev credential for the new instance

Phase 5 — Deploy & Verify infra

5.1 Coolify deployment

  1. Create new Coolify app pointing at the new git repo.
  2. Set all env vars from Phase 1 (DB, R2, GCP TTS, Anthropic, Gemini, NEXT_PUBLIC_APP_URL, BETTER_AUTH_SECRET).
  3. Attach the new domain.
  4. Deploy. The Dockerfile entrypoint runs drizzle-kit push --force on boot, so the schema will be created automatically.

5.2 End-to-end smoke test on the new domain

  1. Log in as admin, create a school + teacher.
  2. Teacher creates a class, enrolls a test student, prints a class shortcode.
  3. Student logs in via /c/<shortcode>; visual-password flow works.
  4. Teacher creates a story → AI plan generates with new prompt branding → prose + questions follow → student records → playback works (R2 + audio proxy route).
  5. Teacher creates a spelling list → words get Gemini images + GCP TTS audio → student plays snowman game.
  6. Attendance + syllabus pages load, file uploads to R2 succeed.
  7. Confirm no calls go to the old Macmillan R2 / DB / TTS project (check network tab + Coolify logs).

Risks & Watch-outs

  • Schema rename migration: renaming af_f_level after the schema is already deployed is a ALTER COLUMN RENAME — safe, but coordinate with the first deployment so it runs against the empty DB rather than retrofitting later.
  • Coolify build cache: Coolify caches Node modules between builds. After cloning, the first build will be slow; subsequent ones fast.
  • R2 public URL format: if you use the Cloudflare public bucket URL rather than a custom domain, audio proxy routes (/api/audio/[...key]) still work — but consider whether the new school wants a vanity domain for media.
  • Forgetting the seed vocab: reading-passage generation will fail with "Target vocabulary IDs not found" if you forget to seed words after schema push. Run the new seed scripts before any teacher tries to use the reading feature.
  • Two codebases diverging: after the fork, any bug you fix in one is on you to manually port to the other. Keep a personal note of fixes that should cross over.
